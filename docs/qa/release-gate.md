# QA, Performance, and Release Gate

Status: G006 scaffold. This defines the final gate; it does not pass while real-device ASR/MT proof goals are failed or pending.

## UI automation scope

mobile-mcp should cover:

- app launch;
- microphone permission prompt path;
- language selection;
- local-only/offline mode state;
- missing offline pack state;
- unsupported translation pair state;
- cloud consent screens;
- cloud disabled/unavailable state;
- transcript display shell.

mobile-mcp is not performance authority. Native instrumentation owns latency, memory, battery, and thermal evidence.


### mobile-mcp scenario harness

The mobile-mcp QA harness is defined by `configs/mobile-mcp-scenarios.json`, documented in `docs/qa/mobile-mcp-automation.md`, and concrete run artifacts are checked by `scripts/validate_mobile_mcp_run_artifacts.py`. Validate the harness with:

```sh
python3 scripts/validate_mobile_mcp_qa.py
python3 scripts/validate_mobile_mcp_run_artifacts.py --run-dir docs/qa/mobile-mcp-samples/android/launch-offline-local-first --allow-sample
```

mobile-mcp evidence is UI-flow QA evidence only. It can cover launch, permission, offline/local-first, missing model, unsupported pair, and explicit cloud-consent flows on iOS/Android real devices, simulators, or emulators. It must not be used as ASR/MT performance proof, battery/thermal proof, legal approval, or a support-claim authority.

## Native performance scope

Required native/device evidence:

- ASR p95 partial latency;
- ASR final latency;
- MT p95 final latency where supported;
- memory high-water mark;
- battery drain over 30 minutes;
- thermal state over 30 minutes;
- audio dropouts;
- model download/interruption rollback;
- no-silent-cloud network invariant.




### Device-lab intake report

The release gate requires `scripts/validate_device_lab_intake_report.py` so every
external G008/G010 evidence drop can leave `docs/benchmarks/device-lab-intake.generated.json`
and `.md` audit artifacts. In the current repository state this report must stay
`BLOCKED` with zero real bundles and zero supported rows.

### Device-lab execution preflight

Real G008/G010 retries must pass `scripts/validate_device_lab_preflight.py` on
the lab machine before collecting evidence. The release gate runs
`python3 scripts/validate_device_lab_preflight.py --allow-missing --expect-blocked`
in the repository planning state to prove the current environment is still
blocked and is not silently converted into support evidence.

## Automated release-gate script

`scripts/check_release_gate.py` must run all scaffold validators, the mobile-mcp QA harness validator, the device-lab kit validator, the single-command evidence-root intake wrapper, the support-matrix generator, telemetry log validator, no-false-support verifier, and generated completion audit before checking Ultragoal status. The gate remains blocked while:

- any non-superseded goal is failed or pending;
- generated support matrix has no real supported rows for required launch scope;
- G008/G010 proof blockers are unresolved;
- G012 final cleanup/review has not passed.

The proof retry decision is generated by:

```sh
python3 scripts/generate_proof_retry_decision.py
python3 scripts/validate_proof_retry_decision.py
```

It writes `docs/benchmarks/proof-retry-decision.generated.json` and `.md`, and
must stay `BLOCKED` in the current repository state. This artifact is a retry
review gate only; it is not ASR/MT benchmark proof, legal approval, or a support
claim.

The gate is intentionally conservative: mock or `not_claimed` evidence can appear in the generated matrix, but cannot become a support claim.

The evidence-root intake step is:

```sh
python3 scripts/validate_device_lab_evidence_root.py --include-samples
```

It is safe before real lab evidence exists: it validates sample bundles, refreshes the generated support matrix, and reports zero supported rows while G008/G010 remain unresolved.

The completion audit is generated by:

```sh
python3 scripts/generate_completion_audit.py
```

It writes `docs/qa/completion-audit.generated.md` and must show `BLOCKED` until G008, G010, and G012 have real proof/review evidence.

## Final completion blockers

The aggregate Ultragoal cannot complete while any of these are true:

- failed real-device ASR proof exists;
- failed real MT/runtime/legal proof exists;
- final verification scripts fail;
- ai-slop-cleaner has not run;
- independent code review is not APPROVE/CLEAR.


## No-false-support guardrail

`scripts/validate_no_false_support_claims.py` is part of the automated release gate while G008/G010 are unresolved. It verifies current candidate configs, sample benchmark results, sample bundles, model manifests, and the generated support matrix remain conservative: no `supported` rows, no sample distribution approval, and no ASR/MT support claim before real evidence bundles pass review.
